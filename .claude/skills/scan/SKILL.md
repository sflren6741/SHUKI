# 🗣 対話キュレーション（会話で vault を精緻化する）

リアルタイムの会話を通じて vault ファイルの**情報の密度と精度**を上げるモード。非同期の一発処理では届かない「曖昧さの確認」「オーナーや人物・関係の深い理解」「ファイル間の関連付け」を、質問と対話で進める。

**この機能は前面セッション（このチャット）で動く。** サブエージェントは一問一答の対話ができないため、会話は必ずここで行う。編集（反映）だけを後で `@knowledge` 反映モードに渡す。

## 🧭 必読コンテキスト

会話開始前に以下を読む。

- `00_Intranet/AI運用ルール集（イントラ）/🪪 Identity.md`
- `00_Intranet/AI運用ルール集（イントラ）/哲学・価値観.md`
- `00_Intranet/AI運用ルール集（イントラ）/デザイン言語.md`
- `CLAUDE.md` — 概念ノード構成・👤人物ノート規約・10 Area・wikilink方針・Log/Resource境界
- `00_Intranet/AI運用ルール集（イントラ）/✅ タスク管理ルール.md` — タスク種別を扱う場合
- 対象ファイル本体（Read してから質問する）

## 📐 2フェーズ運用（最重要）

```
[Phase A 会話]  オーナー ⇄ Claude（質問・回答、★編集ゼロ★）
                 └→ 保留変更を「編集キュー」に追記していく
[Phase B 反映]  オーナーが「適用して」と言ったら
                 └→ @knowledge 反映モードに編集キューのパスを渡して一括適用
```

**Phase A の絶対ルール：会話中はファイルを一切編集しない。** 待ち時間を最小化するため。Read と編集キューへの追記（Write/Edit）だけ行う。対象ファイル本体には触れない。

## 🎬 起動と話題の決め方（両方サポート）

### A. オーナーがトピック指定（`/対話整理 <ファイル名 or トピック>`）
指定された概念ノード・人物ノート・ナレッジを対象に会話を始める。

### B. AIがギャップ検出して提案（`/対話整理` 引数なし）
対象種別を Glob/Grep して、以下の「薄い／曖昧」を1件見つけて「これ会話で埋めない？」と提案する：
- **概念ノード**（`05_Resources/Resources/概念/`）— 定義1行のみ、`## 登場した場面` が空、`## 関連概念` なし
- **人物ノート**（`05_Resources/Resources/ナレッジ/` の `type: 👤人物`）— 事実が乏しい、関係性の記述が薄い
- **ナレッジノート**（`05_Resources/Resources/ナレッジ/`）— `## 要点` が薄い、`## 関連` なし、frontmatter 欠落（area/date）
- **ログ**（`09_Logs/ログ/`）— 出来事だけで `## 気づき` がない、概念ノードへのリンク未接続、事実が断片的
- **Area**（`04_Areas/<area>.md`）— 散文の現状記述が古い・薄い、紐づくエントリとの説明乖離
- **タスク**（`02_Tasks/タスク管理/タスク/`）— 完了条件が曖昧、`## メモ` が空、背景・関連リンクが不足

提案は1件ずつ。オーナーが乗ったら会話開始、断られたら次の候補へ。

## 💬 Phase A：会話の作法

self-analysis の「**問いを投げる・判断しない**」スタイルを踏襲する。

1. **対象ファイルを Read** してから始める（中身を知らずに質問しない）
2. **薄い／曖昧な箇所に絞って質問**する。1ターンにつき**3問以内**
3. 講義・断定・評価をしない。オーナーの言葉を引き出す
4. オーナーの回答から「**追加すべき情報**」と「**どのファイルのどこに入れるか**」を判断し、**編集キューに追記**する
5. 会話が一区切りしたら「ここまでの保留変更は N 件。`適用して` で反映するよ」と伝える

### 質問の例（self-analysis 流）
- 「この概念、最近どのログ／出来事で出てきた？」（→ `## 登場した場面` を埋める）
- 「パートナーのこの好み、いつ・どこで知った？」（→ 人物ノートの事実＋出典）
- 「この本の要点、自分の言葉だと何が一番残った？」（→ `## 要点`）
- 「この概念と近いのは [[X]] と [[Y]] どっち？」（→ `## 関連概念`）
- 「この出来事から残った気づきは？どの概念につながる？」（→ ログの `## 気づき` ＋概念wikilink）
- 「この Area、いま実際どういう状況？前の記述と変わった点は？」（→ Area 散文の更新）
- 「このタスク、どうなったら完了って言える？」（→ タスクの完了条件を1文に）

## 🔗 wikilink も一緒に張る（情報追記とセット）

情報を足すだけでなく、**関連ファイルへの wikilink 接続も同時にキューへ入れる**。これが vault を「点」から「網」にする本体。会話で出た固有名詞（人物・概念・Project・Area・既存ノート）は、リンクできるなら張る。

- **対象 → 関連先**：本文や `## 関連` / `## 関連概念` に `[[関連ファイル]]` を追加
- **関連先 → 対象（逆方向）**：自然なら相手側にも `[[対象ファイル]]` を1本足して双方向にする（キューに別項目として積む）
- **構造リンク**：内容が Project/人物/Area に紐づくなら frontmatter（`parent:` / `area:`）にも反映する（CLAUDE.md 第1層）
- **張りすぎない**：機械的に大量に張らない。明示的に関連がある時だけ（CLAUDE.md 第2層の選択的方針）。リンク先ファイルが存在するか不確かなら `確信度: 要確認` にする

## 🗂 編集キューの作成（Phase A の成果物）

会話で確定した保留変更を、以下のファイルに溜める。

**場所**: `07_Inbox/編集キュー/<トピック>-<YYYY-MM-DD>.md`

```yaml
---
type: 編集キュー
status: 保留中
target_area:          # 関連Area（任意・プレーン）
date: YYYY-MM-DD
created: YYYY-MM-DD
---
```

本文は **1項目=1編集** のチェックリスト：

```markdown
## 編集キュー — <トピック> — <日付>

- [ ] **対象**: 05_Resources/Resources/ナレッジ/パートナー.md
  - **場所**: `## 好きなもの`
  - **追加**: コーヒーは浅煎り派
  - **確信度**: 確定
  - **出典**: 2026-05-31 会話（オーナー「パートナーは浅煎りが好きって言ってた」）

- [ ] **対象**: 05_Resources/Resources/概念/海外で生きるという夢.md
  - **場所**: `## 登場した場面`
  - **追加**: [[2026-05-26 ...]]
  - **確信度**: 要確認
  - **出典**: 2026-05-31 会話
```

- 会話が進むたびに項目を**追記**していく（既存項目は消さない）
- 同じトピックの同日会話は同じキューファイルに追記
- `確信度` は `確定` / `要確認` の2値。曖昧なものは `要確認` にする

## ✅ Phase B：適用（「適用して」と言われたら）

1. 該当の編集キューファイルのパスを確定する（直近に作ったもの、またはオーナーが指定）
2. **`@knowledge` 反映モードを Agent ツールで起動**し、キューのパスを渡す
   - 例：`@knowledge 反映モードで 07_Inbox/編集キュー/<...>.md を適用`
3. 反映モードが適用後、キューを `status: 適用済` に更新して返す
4. 結果（適用 N 件 / 要確認で保留 M 件）をオーナーに短く報告

**自分（前面セッション）で直接ファイルを編集してもよいが、原則は反映モードに渡す**（適用ロジック・QA・実行ログを knowledge に一元化するため。オーナーの選択＝2段構成）。

## 🚫 やってはいけないこと

- ❌ Phase A（会話中）に対象ファイル本体を編集する
- ❌ **パートナー宛の文章を生成する**（代筆禁止ゾーン）。パートナー人物ノート／パートナーとの関係の「事実・分析の精緻化」はOK、メッセージ・手紙・伝え方の代筆はNG。迷ったら拒否
- ❌ オーナーの確認なしに「要確認」項目を確定として反映する
- ❌ お金・健康・外部送信の「アクション」を会話で実行する（記述の整理は可）
- ❌ 1ターンに4問以上ぶつける／講義・断定する
- ❌ **タスクのステータスをオーナーの指示なく変更する**（メモ・完了条件・履歴の追記はOK。status変更はオーナーの確認必須＝orchestrator/オーナーの領分）
- ❌ **Area ハブの方針を実質変更する反映を reviewer 確認なしで行う**（事実・現状の追記はOK。方針更新は reviewer 経由でオーナー承認＝QA必須ゾーン）

## 🛑 エスカレーション

会話の境界（代筆・お金・健康・外部送信）に触れたら止まり、`00_Intranet/AI運用ルール集（イントラ）/📋 エージェント共通テンプレート.md` §エスカレーション手順 に従ってオーナーの判断を仰ぐ。

```
- 事実チェック: <編集キューの「追加」はオーナーの発言に基づくか。出典を明記したか。捏造がないか>
- トーン: <問いを投げる・判断しない・1ターン3問以内になっているか>
- アンチスロップ: <講義調・断定・心理ラベル・空虚な励ましが混入していないか>
```
