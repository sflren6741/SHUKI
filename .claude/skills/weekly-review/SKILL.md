# 📅 週次レビュー（対話型）

オーナーの1週間を振り返り、`07_Logs/ログ/週次レビュー/<YYYY-MM-DD>.md` にレポートを作る。
**前面セッションで対話しながら**作るのが従来の無人生成との違い。直近ログに関係・健康などセンシティブな内容が含まれることがあるため、断定せずオーナーに確認しながら進める。

## 🧭 必読コンテキスト

- `00_Intranet/AI運用ルール集（イントラ）/🪪 Identity.md`
- `00_Intranet/AI運用ルール集（イントラ）/哲学・価値観.md`
- `00_Intranet/AI運用ルール集（イントラ）/デザイン言語.md`

## 📊 入力ソース

| ソース | パス | 内容 |
|---|---|---|
| デジタル活動サマリー | `<WORKSPACE>\weekly_summary.md` | 外部データ収集スクリプトが生成するサマリー（各自で用意） |
| Claude使用状況 | `07_Logs/AI対話履歴/<期間終了日>.md` | セッション数・トークン使用量・ツール使用Top・推定コスト |
| タスクDB | `04_Tasks/タスク管理/タスク/*.md` | 今週完了・作業中・保留のタスク |
| ログ | `07_Logs/ログ/` 直近7日 | 行動ログ・振り返り |

> `weekly_summary.md` が無い場合は該当セクションを「データなし」でスキップ。

## 🔄 フロー

### Step 1: データ前処理

データソースは2系統に分かれる：

| データ | 取得方法 | 自動/手動 |
|---|---|---|
| デジタル活動履歴 | `<WORKSPACE>\vault-scripts\sync_weekly_history.py` 等 | **自動**（スケジュール設定次第） |
| 外部サービスデータ | 各サービスのエクスポート機能 | **手動**（オーナーがエクスポートした時のみ） |
| Claude 使用状況 | `<WORKSPACE>\vault-scripts\analyze_claude_usage.py` | このスキルで実行可 |

手順：
1. `weekly_summary.md` の存在を確認する（スケジュールタスクが事前に生成済みの前提）
2. 外部サービスデータを含めたい場合のみ、オーナーがエクスポート済みであることを確認し対応スクリプトを実行：
```
python <WORKSPACE>\vault-scripts\sync_google_takeout.py    # 外部サービスデータ（手動エクスポート前提）
python <WORKSPACE>\vault-scripts\resolve_places.py         # 場所解決
python <WORKSPACE>\vault-scripts\generate_summaries.py     # 活動サマリー
python <WORKSPACE>\vault-scripts\analyze_claude_usage.py   # Claude使用状況
```
3. データが無ければ当該セクションは「データなし（エクスポート未実施）」と明記してスキップ

### Step 2: 入力を読む

上記「入力ソース」の各ファイルを Read する。`weekly_summary.md` が無い／古い場合は「データなし」でスキップ。

### Step 3: 対話で組み立てる（このスキルの核心）

- ログにセンシティブな内容（関係の危機・健康・センシティブな話題等）があれば、**機械的に要約せずオーナーに「これ、どう扱う？」と確認**してから書く
- データがある場合は**数字をそのまま並べず、データの裏にある行動・関心・変化を読み解く**
- 不明・両義的な点はオーナーに質問する（1ターン3問以内）
- タスク棚卸が必要そうなら `/review-tasks` を案内（このスキルでは深追いしない）

### Step 4: レポート作成

`07_Logs/ログ/週次レビュー/<YYYY-MM-DD>.md` に Write。§1〜§8 の構成で作成する（下記「出力セクション構成」参照）。frontmatter に `date`（期間開始日）/ `created` を付ける。

作成前に：
1. `@reviewer` にQA週次集計を依頼（§8 に取り込む）
2. `@infrastructure` に静かな故障チェックを依頼（§7 に取り込む）
3. §1〜§6 を自分で生成し、§7・§8 を取り込んで1ページにまとめる

## 📋 出力セクション構成（§1〜§8）

週次レビューファイルの構成。frontmatter 直後から以下の順で書く。

```markdown
## §1 今週の最重要3つ

- [タスク名] — 進捗・止まっている理由（あれば）

## §2 数字サマリ

- 完了タスク数 / 起票タスク数
- 期限切れタスク数
- レビュアーから差し戻し件数

## §3 各Areaの動き

（自分の `05_Areas/` 配下のArea名に合わせて記載。Area数・名称は自由）

- `<Area 1>`: 今週のハイライト・課題
- `<Area 2>`: 今週のハイライト・課題
- ...

## §4 来週の判断ポイント

- 期限が近い意思決定リスト
- セカンドオピニオン推奨項目

## §5 デジタル活動サマリ

（weekly_summary.md がある場合のみ。データなしの場合は「データなし」と明記）

1〜3段落の散文で書く。数字の羅列にしない。データの裏にある行動・関心・変化を読み解く。
前週と比較して変化・継続性があれば言及する。

## §6 Claude使用状況

（07_Logs/AI対話履歴/<期間終了日>.md がある場合のみ）

使い方パターン＋使用量を3〜5行に要約。

## §7 インフラ報告

（@infrastructure からの静かな故障チェック結果）

## §8 QAレポート

（@reviewer からのQA週次集計）
```

## 🚫 やってはいけないこと

- ❌ センシティブなログ（個人間コミュニケーション等）をオーナーの確認なく断定的に解釈・要約する
- ❌ センシティブな相手への文章を代筆する（代筆禁止）
- ❌ 数字の羅列で終える（行動の解釈を伴わせる。ただし推測は「かもしれない」）
- ❌ お金・健康の「アクション」を実行する（記述・提案のみ）

- 関係の話題が出たら該当する人物ノートへ

## 🔄 セルフQAの3行

```
- 事実チェック: <数値・日付・出来事はソースに実在するか。推測は仮説と明示したか>
- トーン: <散文分析・断定回避・オーナーの文体に沿うか>
- アンチスロップ: <数字羅列・励まし・心理ラベルがないか>
```
