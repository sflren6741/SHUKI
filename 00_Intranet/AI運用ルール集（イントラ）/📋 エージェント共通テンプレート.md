## エスカレーション手順

判断の境界線を超えた、または完了条件を満たせないと判断した瞬間に止まり、以下を**この順番で**実行する。これは「失敗」ではなく**正しい完了形**。

1. **引継ぎノートを書く** — タスクのコメント欄に「ここまでやったこと / 選択肢と根拠 / 推奨案 / 次の一手」を記録
2. **`agent` フィールドを `人間（オーナー）` に変更する**（frontmatter）
3. **`ai_judgment_ok` を `false` に変更する**（frontmatter）
4. 実行を停止し、オーナーの判断を待つ

詳細手順: `00_Intranet/AI運用ルール集（イントラ）/✅ タスク管理ルール.md` §3.6

## 実行ログ YAMLテンプレート

Write ツールを持つエージェントは、タスク終了時（完了・エラー問わず）に `00_Intranet/agent-runs/` へ実行ログを書く。

ファイル名：`YYYY-MM-DD_HHmm_<agent-name>_<slug>.md`

```yaml
---
agent: <agent-name>
trigger: manual | scheduled
prompt: "<受け取ったプロンプト概要>"
started: YYYY-MM-DDTHH:mm:ss
finished: YYYY-MM-DDTHH:mm:ss
result: success | partial | error
error_message: ""
summary: "<何をしたか1行>"
---
```

`session_limit` の場合はPS1スクリプト層が検知して記録する（エージェント自身は書かなくてよい）。

## YAML（frontmatter）出力ルール

frontmatter を書くすべてのエージェントは、以下を**必ず守る**。破ると Obsidian / パーサが「mapping values are not allowed here」等で frontmatter 全体を壊す。

1. **独立したメタデータ（`agent` / `ai_judgment_ok` / `status` / `area` 等）は、必ず1項目1行に改行して出力する。** 複数のメタデータを `memo:` などの1つの値の中にまとめて埋め込まない。

   ```yaml
   # ❌ NG（1つの値に詰め込み、コロンが値の途中に出て壊れる）
   memo: タスク化した。agent: infrastructure、ai_judgment_ok: false

   # ✅ OK（個別行に分離）
   memo: "タスク化した。"
   agent: infrastructure
   ai_judgment_ok: "false"
   ```

2. **値の文字列内に `:`（コロン）・`false`/`true`・絵文字・`#`・`「」` 等が含まれる場合は、値全体を必ず `"` で囲む。**

   ```yaml
   # ❌ NG
   status: ✅振り分け済
   ai_judgment_ok: false
   memo: A案（type:アイデア）とB案を比較

   # ✅ OK
   status: "✅振り分け済"
   ai_judgment_ok: "false"
   memo: "A案（type:アイデア）とB案を比較"
   ```

   - `true`/`false` を**真偽値として**使いたい場合のみクォート無し（例: 機械処理する boolean フラグ）。文字列メモの一部としての `false` は必ずクォートする。

3. 複数行テキストは `|`（リテラルブロック）を使い、各行をインデントする。

書き終えたら frontmatter を一度見直し、各メタデータが個別行になっているか・`:` や絵文字を含む値がクォートされているかを確認してから保存する。
